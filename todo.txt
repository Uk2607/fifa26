Updated Business Logic
To fix the points table leakage, we need to introduce a strict conditional check based on the match state.
- UPCOMING State: The match acts as a placeholder. It must be completely ignored by the points table calculator. Total matches played, wins, losses, draws, and points remain unchanged.
- OPEN State (or Live/Completed): The match is active or concluded. Its data contributes normally to the team statistics and points table.

UI / UX Requirements (Group Pop-up Toggle)
To allow users to switch states easily, implement a toggle directly inside the Group Pop-up.
Behavioral Flow:
- User clicks on a specific match inside the group view to open the Group Pop-up.
- Inside the pop-up, a prominent Status Toggle Switch (or Radio/Dropdown) is displayed.
- Toggle Options:
-- Upcoming (Default for future matches)
-- Open (Ready for scoring/active)
- On Click / Change:
-- Update the status property of that specific match in the state.
-- Trigger a re-calculation of the points table (calculatePointsTable).

Expected Behavior Check
- Toggle set to UPCOMING: Points table ignores the match. (Matches Played count does not increase).
- Toggle switched to OPEN: Points table instantly updates to include the match data.

Suggestion: For three way slider to lock the result, only if the the default value is not 'locked'. For other two status we can do toggle the three way slider.
Instead of a simple binary toggle, the Group Pop-up will feature a Three-Way Slider / Segmented Control (Upcoming ↔ Open ↔ Locked).
- If the default status is LOCKED: The slider is disabled/frozen. The user cannot accidentally toggle it back to Open or Upcoming.
- If the default status is UPCOMING or OPEN: The user can freely toggle back and forth between these three states.
